# Development Workflow

## 这份文档管什么

这份文档不是在定义“通讯协议产品化规范”的输出格式。

它定义的是：

> **这个库自己在被开发时，必须按什么流程推进。**

## 总流程

这个库后续开发，一律按下面顺序走：

1. 需求澄清
2. 开发任务拆解
3. MRD
4. PRD
5. BMaD 开发

但在这 5 步之前，还有一个更上层的固定约束：

`Global Goal`

也就是：

`Global Goal -> 需求澄清 -> 开发任务拆解 -> MRD -> PRD -> BMaD 开发`

## 每一步必须留下什么

### 1. 需求澄清

目标：

- 先把这轮到底要解决什么说清
- 把不做什么也说清
- 防止一边问问题，一边已经偷偷开始设计方案

产物：

- 一份澄清纪要文档

### 2. 开发任务拆解

目标：

- 把问题拆成可推进的任务块
- 明确先后顺序
- 区分主线、支线、以后再做

产物：

- 一份任务拆解文档

### 3. MRD

目标：

- 把这轮要交付的东西写成正式定义
- 说清楚目标用户、核心问题、边界、验收标准

产物：

- 一份 MRD 文档

### 4. PRD

目标：

- 把第一版产品要求写清
- 把 workflow、状态、命令、阻断规则定义清楚
- 给开发阶段提供明确边界

产物：

- 一份 PRD 文档

### 5. BMaD 开发

目标：

- 按前面已经确认过的文档进入开发
- 不在开发阶段重新发明目标

产物：

- 一份开发执行文档
- 对应代码 / 配置 / 测试 / 验证结果

## 过程规则

### 规则 1：聊天不算落盘

只在对话里说过，不算完成。

只要形成共识，就要进入当前 repo 文档。

### 规则 2：澄清不等于切阶段

用户继续追问、补充、发散、想到新点子，都默认还在当前阶段。

除非明确确认切换，否则不能自动从：

- 需求澄清 跳到 任务拆解
- 任务拆解 跳到 MRD
- MRD 跳到 PRD
- PRD 跳到 BMaD 开发

### 规则 3：先文档，后开发

这个库当前阶段的要求很明确：

先把每个阶段该有的文档沉淀下来，再进入实现。

不能因为“感觉已经聊明白了”就直接跳代码。

### 规则 4：局部问题不能吃掉全局目标

每次只要讨论进入细节，都要能回答：

- 现在的 `Global Goal` 是什么
- 当前处在哪个阶段
- 这一段讨论是在推进主线，还是只是补充信息

如果答不上来，说明已经开始漂了。
