# Development Step Breakdown

## 这份文档的作用

这份文档是这个项目的 **开发步骤拆解文档**。

它回答的是：

> 这个项目接下来按什么步骤推进，每一步要产出什么，什么情况下才能进入下一步。

## 总体开发顺序

当前固定开发顺序为：

1. `Global Goal`
2. `需求澄清`
3. `开发任务拆解`
4. `MRD`
5. `PRD`
6. `BMaD 开发`

这里的重点不是“把步骤列出来”，而是：

每一步都必须变成文档和状态，而不是只停留在对话里。

## Step 1：Global Goal

### 目的

先锁这个库的长期主线，不让后面讨论越聊越变。

### 当前状态

已完成。

### 当前产物

- `docs/global-goal.md`

### 通过标准

- 能一句话说明这个项目在做什么
- 能说明这个项目不在做什么

## Step 2：需求澄清

### 目的

把“当前到底要解决什么问题”说清楚。

### 当前状态

已完成并单独沉淀。

### 当前产物

- `docs/requirements-clarification.md`

### 通过标准

- 当前痛点明确
- 项目目标明确
- 非目标明确
- 可以进入任务拆解

## Step 3：开发任务拆解

### 目的

把这个产品拆成可以逐步推进的工作块，而不是直接跳实现。

### 当前状态

已开始并沉淀。

### 当前产物

- `docs/task-breakdown.md`
- `docs/development-step-breakdown.md`（本文件）

### 当前拆解结果

目前已经拆出 8 个 workstream：

1. Workflow Canonical Spec
2. Document Contract
3. State Model
4. Storage Design
5. CLI Workflow Commands
6. Guard Rules
7. Testing Strategy
8. Phase 1 Scope Boundary

### 通过标准

- 第一版范围明确
- 任务块边界明确
- 开发顺序明确
- 可以进入 MRD

## Step 4：MRD

### 目的

把第一版产品正式写成一份可执行的定义文档。

### 当前状态

已开始草稿。

### 需要产物

- 一份正式 MRD

### 预计应包含

- 产品目标
- 核心问题
- 第一版用户
- 核心 workflow
- 边界控制机制
- 非目标
- 验收标准

### 进入条件

- 需求澄清通过
- 任务拆解通过

## Step 5：PRD

### 目的

把第一版产品要求写成正式定义，锁住实现边界。

### 当前状态

已有 `pre-draft`，但尚未正式进入。

### 需要产物

- 一份正式 PRD

### 进入条件

- MRD 通过

## Step 6：BMaD 开发

### 目的

在 PRD 已确认后，再进入真正的开发实现。

### 当前状态

未开始。

### 需要产物

- 开发执行文档
- 对应代码与测试
- 验证结果

### 进入条件

- PRD 通过

## 当前推荐的推进顺序

按现在状态，接下来只做下面这个顺序：

1. 固定需求澄清文档
2. 固定开发步骤拆解文档
3. 写 MRD
4. 写 PRD
5. 再进入 BMaD 开发

## 当前明确不允许的跳跃

下面这些现在都不应该直接发生：

- 从澄清直接跳代码
- 从任务拆解直接跳平台接入
- 在 PRD 还没写时开始实现状态机
- 在边界没锁死时扩展集成层

## 这一阶段完成后意味着什么

当这份文档和任务拆解文档都稳定后，说明这个项目第一次真正进入了：

> **不是靠聊天推动，而是靠 workflow 文档推动。**
