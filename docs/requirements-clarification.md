# Requirements Clarification

## 这份文档的作用

这份文档是这个项目的 **需求澄清文档**。

它回答的是：

> 这个库现在到底要开发什么，为什么要开发它，当前已经澄清到什么程度。

这不是实现文档，也不是 MRD。

## 项目背景

当前痛点不是“AI 不够聪明”，而是：

- 对话一长，AI 很容易顺着局部问题往下走
- 澄清问题经常被误当成切阶段
- 你和 AI 都可能被新想法带偏
- 最后产出很多局部内容，但丢掉全局主线

所以这个项目要做的，不是普通助手，而是一个 **AI 协作里的防漂移控制系统**。

## 当前需求一句话

把 AI 协作从“容易随对话漂移的聊天过程”，做成一个 **有边界控制、有直接 workflow、不会偷偷切阶段** 的强约束系统。

## Global Goal

把 AI 协作里的“防漂移”做成一个强约束控制系统，优先保证：

- `Goal` 不变
- `阶段` 不乱切
- `澄清` 不偷偷改 `Next`

并把这套控制能力嵌进：

`需求澄清 -> 开发任务拆解 -> MRD -> BMaD 开发`

## 当前已经明确的核心要求

### 1. 要有边界控制

系统必须能明确判断：

- 当前在哪个阶段
- 哪些动作只是继续当前阶段
- 哪些动作是在申请切阶段
- 哪些动作必须被 block

### 2. 要有直接 workflow

这个项目不是停留在理念层。

最后一定要有能直接驱动流程的 workflow，而不是继续靠聊天记忆推进。

### 3. 不能漂移

这里的“不能漂移”至少包括：

- 不能偷偷改 `Goal`
- 不能因为追问就自动切阶段
- 不能因为灵感插入就替换当前主线
- 不能在没有文档沉淀的情况下直接跳实现

### 4. 先文档，后开发

你已经明确提出：

- 需求澄清先沉淀成文档
- 开发步骤拆解先沉淀成文档
- 后面再进入 MRD 和开发

也就是说，聊天本身不算完成，必须落盘到 repo。

## 当前已确认的开发流程

这个库自身的开发流程固定为：

`Global Goal -> 需求澄清 -> 开发任务拆解 -> MRD -> BMaD 开发`

注意：

这里是在说 **开发这个产品本身的流程**，
不是这个产品未来提供给别的项目的协议输出格式。

## 已确认技术方向

技术方向已通过：

- `方案 B：状态机驱动的 Workflow Engine`

当前建议技术栈：

- Python 3.12+
- `uv`
- Markdown
- YAML frontmatter
- JSON
- SQLite
- `typer`
- `pydantic`
- `pytest`

## 当前阶段判断

需求澄清已经基本收口。

当前可以认为：

- 需求澄清文档：需要正式存在
- 开发步骤拆解文档：需要正式存在
- MRD：还没开始
- BMaD 开发：还没开始

## 当前不做的事

这一轮明确不做：

- 直接写 runtime / adapter / 平台集成
- 先上 Web UI
- 先做多人协作权限
- 先做 Hermes / Discord / Cloud DROID 深接入

## 这份文档通过的标准

当下面几件事都成立时，这份澄清文档就算通过：

1. 项目目标明确
2. 当前阶段明确
3. 非目标明确
4. 技术方向明确
5. 可以无歧义进入“开发步骤拆解”
