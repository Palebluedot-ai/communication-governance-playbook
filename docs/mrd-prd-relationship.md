# MRD PRD Relationship

## 先说明一下

你上一条里写的是：

`MRD、PRD 和 MRD 之间是什么关系`

我先按你想问的是：

> **MRD 和 PRD 分别是什么，它们之间是什么上下游关系。**

来落这份文档。

如果你其实想补的是 `BRD`，我们下一轮再把 `BRD / MRD / PRD` 三层完整补上。

## 一句话关系

简单说：

- `MRD` 解决的是：**为什么做、给谁做、解决什么问题**
- `PRD` 解决的是：**具体做什么、做到什么程度、怎么验收**

所以关系不是并列，而是：

> `MRD -> PRD -> BMaD 开发`

## MRD 是什么

`MRD` = `Market Requirements Document`

在这个项目里，不一定是“外部市场”那种狭义市场，更准确说是：

> **需求与价值定义文档**

它要回答：

- 这个产品为什么值得做
- 当前真正的痛点是什么
- 谁最需要它
- 第一版到底解决哪一段问题
- 不解决什么

### 对这个项目来说

这个项目的 `MRD` 重点会放在：

- AI 协作为什么会漂移
- 这种漂移造成什么成本
- 为什么要做成强约束 workflow
- 第一版用户是谁
- 第一版只打哪一个最小切口

## PRD 是什么

`PRD` = `Product Requirements Document`

它是在 `MRD` 已经通过后，继续往下定义：

> **这个产品第一版具体要长成什么样。**

它要回答：

- 有哪些核心对象和状态
- 有哪些命令、界面、输入输出
- 哪些规则必须阻断
- 哪些属于第一版，哪些不是
- 最终怎么验收

### 对这个项目来说

这个项目的 `PRD` 重点会放在：

- workflow 状态机长什么样
- phase / transition / guard 怎么定义
- CLI 命令有哪些
- 文档契约有哪些字段
- SQLite / JSON / Markdown 各管什么

## 为什么不能跳过 MRD 直接写 PRD

因为这个项目最容易犯的错就是：

- 还没把“为什么做、先打什么”讲清
- 就开始写状态机、命令、schema、guard

这会直接重演你之前最不满意的那种情况：

> 一直往下做局部实现，但离最开始的目标越来越远。

所以这个项目里：

- `MRD` 先锁方向
- `PRD` 再锁实现边界

## 在你这个项目里的推荐顺序

当前推荐顺序固定为：

1. `Global Goal`
2. `需求澄清`
3. `开发任务拆解`
4. `MRD`
5. `PRD`
6. `BMaD 开发`

## 每份文档分别产出什么

### MRD 产出

- 产品目标
- 核心问题
- 第一版目标用户
- 第一版最小价值
- 非目标
- 成功标准

### PRD 产出

- 核心 workflow
- 状态模型
- 命令模型
- 文档契约
- 阻断规则
- 存储设计
- 验收规则

## 这次应该先做哪一份

按当前阶段，应该先做：

- `MRD`

不是先把 `PRD` 写满。

`PRD` 可以先建一个骨架，但正式内容应建立在 `MRD` 通过之后。
